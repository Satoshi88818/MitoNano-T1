Agent–Field Coupling

Wires Agent/Engine to ConcentrationField, the next increment after building the two pieces separately: agents now release payload into a shared diffusing field and the whole system's mass is tracked as one invariant, instead of two disconnected demos that happened to both work.

Status: compiles clean, zero warnings. 10 new/changed tests (6 in agent.rs, 4 in coupling.rs), all passing. Full crate: 30/30 tests passing in ~2s.

Design

Two first-order decay pathways, both proportional to remaining payload:

k_elim — true elimination. Payload leaves the system entirely (metabolized, cleared). This is the only pathway the original field-free Agent::step applies.

k_release — release into the environment. Payload leaves the agent but is deposited into the field, not lost.

Agent::step_coupled splits the loss between the two exactly, not approximately: for two simultaneous first-order processes with rates k_release and k_elim acting on the same quantity, the fraction of total loss attributable to each is constant at every instant — k_release/(k_release+k_elim) and k_elim/(k_release+k_elim) respectively. This falls straight out of solving dC/dt = -(k_release+k_elim)·C and integrating each rate's own contribution over the same interval, so released = total_loss × k_release/(k_release+k_elim) is exact algebra, computed with the same closed-form exponential update used everywhere else in this crate (no new numerical approximation introduced).

CoupledEngine ties it together: each tick, every agent moves and releases payload via step_coupled; the released amount is deposited into the field at the agent's new position; then the field takes one diffusion step.

The invariant tests

Zero elimination → exact mass conservation. With k_elim=0, combined mass (all agent payload + all field mass) must stay exactly constant across the whole run, to 1e-9 relative tolerance, at every recorded tick — not just start and end. This follows from two already-proven facts composing correctly: step_coupled with k_elim=0 releases the agent's entire loss (proven in agent.rs), and field.step conserves deposited mass exactly (proven in concentration.rs). If the coupling wiring were wrong — wrong deposit amount, deposit skipped some ticks, double-counting — this test would catch it immediately.

Both pathways active → exact closed-form elimination curve. With k_release>0 and k_elim>0, combined mass is not conserved (elimination genuinely removes mass) — but it must follow C0 · (1 − (k_elim/k_total)·(1 − e^(−k_total·t))) exactly, derived from the same proportional-partition argument. Checked to 1e-6 relative tolerance at every tick over a 600-step run with 25 agents.

Field actually receives mass — a basic wiring sanity check independent of the exact-conservation math: field mass starts at zero and is provably nonzero after agents release into it.

No pathways active → no transfer at all — a no-op control case: field mass stays exactly zero and agent payload stays exactly constant throughout.

Unlike the concentration-field build (where a real convergence issue showed up and had to be diagnosed), these four passed on the first run — expected, since they check exact algebraic identities rather than a numerical approximation converging toward a continuous solution.

What's deliberately not here yet

Agents don't yet sense the field (no feedback loop — release is one-directional, agent→field). Trilinear interpolation on sample() is still nearest-cell. Both are natural next steps once this coupling is confirmed solid.

This is the only pathway `step` (the //! original, field-free API) applies. //! - `k_release`: release into the environment -- payload leaves the //! agent but is deposited into a `ConcentrationField`, not lost from //! the system. Shared by /// both `step` and `step_coupled` so the two can't drift apart in how /// motion is handled. fn advance_position<R: Rng + ?Sized>(&mut self, params: &AgentParams, bounds: &Bounds, dt: f64, rng: &mut R) { let mut new_pos = [0.0; 3]; for i in 0..3 { let drift_i = params.drift[i]; let sigma = (2.0 * params.diffusion_coeff).sqrt(); new_pos[i] = em_step( self.position[i], 0.0, dt, |_x, _t| drift_i, |_x, _t| sigma, rng, ); } self.position = bounds.reflect(new_pos); } /// Advances the agent by one tick with no field coupling: motion plus /// exact exponential decay of payload via `k_elim` alone (`k_release` /// is ignored, since with no field there's nowhere for released /// payload to go). pub fn step<R: Rng + ?Sized>(&mut self, params: &AgentParams, bounds: &Bounds, dt: f64, rng: &mut R) { self.advance_position(params, bounds, dt, rng); self.payload *= (-params.k_elim * dt).exp(); } /// Advances the agent by one tick with field coupling: motion, then /// payload decay split exactly between release (returned, for the /// caller to deposit into a field) and true elimination (silently /// lost, same as `step`). /// /// The split is *exact*, not approximate: for two simultaneous /// first-order processes with rates k_release and k_elim acting on the /// same payload, the fraction of total mass lost to each process is /// constant at every instant (k_release / (k_release+k_elim) and /// k_elim / (k_release+k_elim) respectively) -- this falls straight /// out of solving dC/dt = -(k_release+k_elim)*C and integrating each /// process's own rate contribution over the same interval. So /// `released = total_loss * k_release/(k_release+k_elim)` is the exact /// analytical answer, not a numerical approximation, and the payload /// update itself uses the same closed-form exponential as `step`. pub fn step_coupled<R: Rng + ?Sized>( &mut self, params: &AgentParams, bounds: &Bounds, dt: f64, rng: &mut R, ) -> f64 { self.advance_position(params, bounds, dt, rng); let k_total = params.k_release + params.k_elim; if k_total <= 0.0 { return 0.0; } let payload_before = self.payload; let payload_after = payload_before * (-k_total * dt).exp(); let total_loss = payload_before - payload_after; let released = total_loss * (params.k_release / k_total); self.payload = payload_after; released } } #[cfg(test)] mod tests { use super::*; use rand::SeedableRng; use rand::rngs::StdRng; #[test] fn payload_decays_exactly_per_first_order_kinetics() { let bounds = Bounds::new([-1000.0, -1000.0, -1000.0], [1000.0, 1000.0, 1000.0]);
