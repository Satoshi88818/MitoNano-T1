Actually Using SpatialIndex: Agent-Agent Payload Exchange

The payoff for building SpatialIndex: nearby agents now exchange payload with each other (a lumped model of local contact/transfer), found via the spatial index instead of an O(n²) all-pairs scan every tick.

Status: compiles clean, zero warnings. 4 new tests, all passing. Full crate: 47/47 tests passing in ~1.75s.

Two real bugs caught by the compiler during this build, worth naming honestly since they're the kind of thing that "looks done" if you don't actually run the toolchain:

A Rust borrow-checker error in last turn's SpatialIndex::rebuild (immutable + mutable borrow of self in one expression) — fixed by hoisting the index computation out.

An ambiguous numeric type error in this turn's test ((-2.0 * 0.3 * t_final).exp() where t_final had no type annotation) — fixed by annotating t_final: f64.

Neither is exotic; both are exactly the kind of thing that never shows up in prose specs (like the original repo) and always shows up the moment code actually has to compile.

Design

exchange_pair uses the exact closed-form solution, not an explicit-Euler step: for two agents exchanging payload at rate k_exchange, the system is the linear ODE d(Pi)/dt = -k(Pi-Pj), d(Pj)/dt = +k(Pi-Pj), whose exact solution has the pair's sum invariant and the difference decaying as (Pi-Pj)(t) = (Pi-Pj)(0)·exp(-2kt). Using this directly means the update is unconditionally stable — no CFL-style restriction on dt, unlike ConcentrationField::step's explicit FTCS scheme — and exactly conserves the pair's sum by construction.

step_agent_interactions uses SpatialIndex::neighbors_within to find candidate pairs for each agent, filters to j > i (process each pair once), and sorts neighbor indices before processing — this is what makes the result reproducible independent of the index's internal bucket traversal order, and what makes the brute-force comparison below a fair, order-matched test.

step_agent_interactions_brute_force is a reference-only O(n²) implementation with the identical pairwise rule and identical processing order, kept purely to validate the index-based path against.

The tests

two_agent_exchange_matches_analytical_two_compartment_solution — two stationary agents, always in range, checked against the exact closed-form two-compartment solution over 500 ticks, plus sum-conservation at every single tick.

many_body_total_payload_exactly_conserved_via_spatial_index — 60 moving agents, changing neighbor sets every tick as the index rebuilds, total system payload exactly conserved (1e-9) at every tick over 300 steps.

spatial_index_matches_brute_force_reference_across_full_simulation — the headline test. Two simulations from identical initial conditions and identical RNG-driven motion (positions confirmed bit-identical at every tick as a precondition), one using the spatial index and one using brute-force O(n²) search. Payload trajectories must match to <1e-9 at every tick, every one of 70 agents, across 150 ticks — if the index ever found a different pair set than brute force, this would diverge immediately. It didn't.

zero_radius_means_no_exchange_at_all — control case: radius 0.0 means no pairs are ever found, payloads stay exactly at their initial values despite active motion and repeated rebuilds.

Test 3 is the one that actually validates "SpatialIndex is being used correctly" in a dynamic simulation — the orchestration.rs tests from last time validated the index in isolation (against brute-force queries), but this validates it wired into a real, evolving system where getting the neighbor set wrong would visibly corrupt the physics, not just fail a unit check.

//! Agent-agent local payload exchange: nearby agents equilibrate payload //! toward each other (a lumped model of, e.g., direct contact transfer or //! very-short-range diffusion between particles), using `SpatialIndex` to //! find interacting pairs instead of an O(n^2) all-pairs scan every tick. //! //! This is the actual payoff for having built `SpatialIndex`: without it, //! `step_agent_interactions_brute_force` (kept here only as a test //! reference) is what every tick would cost -- fine for tens of agents, //! prohibitive for the hundreds of thousands this crate is nominally //! aimed at. use crate::agent::{Agent, AgentParams}; use crate::environment::Bounds; use crate::orchestration::SpatialIndex; use rand::Rng; #[derive(Debug, Clone, Copy)] pub struct InteractionParams { pub radius: f64, pub k_exchange: f64, } /// Exact pairwise relaxation toward equal payload. For two agents /// exchanging at rate `k_exchange`, the system is the linear ODE /// d(Pi)/dt = -k*(Pi-Pj), d(Pj)/dt = +k*(Pi-Pj) /// whose exact solution is: the sum Pi+Pj is invariant, and the /// difference decays as (Pi-Pj)(t) = (Pi-Pj)(0)*exp(-2*k*t). Using this /// closed form directly (rather than an explicit-Euler flux*dt step) /// means the update is *unconditionally stable* -- no CFL-style /// restriction on `dt` the way `ConcentrationField::step` needs one -- /// and exactly conserves the pair's sum by construction, to floating /// point precision. pub fn exchange_pair(agents: &mut [Agent], i: usize, j: usize, k_exchange: f64, dt: f64) { let sum = agents[i].payload + agents[j].payload; let diff = agents[i].payload - agents[j].payload; let diff_new = diff * (-2.0 * k_exchange * dt).exp(); agents[i].payload = (sum + diff_new) * 0.5; agents[j].payload = (sum - diff_new) * 0.5; } /// Runs one tick of pairwise exchange using the spatial index to find /// candidate neighbor pairs. For each agent i, neighbor indices j>i /// within `params.radius` are sorted before processing -- this is what /// makes results reproducible independent of the index's internal bucket /// traversal order, and is what lets the brute-force cross-validation /// test below expect a genuinely fair, order-matched comparison. /// /// Each pair is processed with whatever payload values are current at /// that moment (sequential relaxation, not a simultaneous many-body /// solve) -- a deliberate, simple choice: it's still exactly /// sum-conserving regardless of how many overlapping pairs an agent /// participates in or what order they're processed in, since each /// individual pairwise operation only ever redistributes mass between /// the two agents it touches. pub fn step_agent_interactions(agents: &mut [Agent], index: &SpatialIndex, params: &InteractionParams, dt: f64) { for i in 0..agents.len() { let query = agents[i].position; let mut neighbors: Vec<usize> = index .neighbors_within(agents, query, params.radius) .into_iter() .filter(|&j| j > i) .collect(); neighbors.sort_unstable(); for j in neighbors { exchange_pair(agents, i, j, params.k_exchange, dt); } } } /// Reference brute-force implementation: identical pairwise update rule /// and identical (i, then j>i, in increasing order) processing order, but /// finds neighbor pairs by scanning all O(n^2) pairs directly instead of /// using the spatial index. Used only for testing -- this is the ground /// truth `step_agent_interactions` is checked against, to confirm the /// index is finding the *same* interacting pairs a full scan would, not /// just that whatever pairs it finds happen to conserve mass (which is /// true by construction regardless of whether the pairs are correct). pub fn step_agent_interactions_brute_force(agents: &mut [Agent], params: &InteractionParams, dt: f64) { let n = agents.len(); let r2 = params.radius * params.radius; for i in 0..n { for j in (i + 1)..n { let pi = agents[i].position; let pj = agents[j].position; let d2 = (pi[0] - pj[0]).powi(2) + (pi[1] - pj[1]).powi(2) + (pi[2] - pj[2]).powi(2); if d2 <= r2 { exchange_pair(agents, i, j, params.k_exchange, dt); } } } } #[derive(Debug, Clone, Copy)] pub struct InteractionTickStats { pub tick: u64, pub total_payload: f64, } /// Ties motion, the spatial index, and pairwise exchange into one tick /// loop: agents move, the index is rebuilt from their new positions, then /// nearby agents exchange payload. pub struct InteractionEngine { pub agents: Vec<Agent>, /// Only `drift` and `diffusion_coeff` are used here; `k_elim` and /// `k_release` are ignored (this engine models pure motion + local /// agent-agent exchange, no field/elimination -- see `coupling.rs` /// for that side of the simulation). pub motion_params: AgentParams, pub bounds: Bounds, pub index: SpatialIndex, pub interaction: InteractionParams, pub dt: f64, } impl InteractionEngine { pub fn new( agents: Vec<Agent>, motion_params: AgentParams, bounds: Bounds, index: SpatialIndex, interaction: InteractionParams, dt: f64, ) -> Self { Self { agents, motion_params, bounds, index, interaction, dt } } fn total_payload(&self) -> f64 { self.agents.iter().map(|a| a.payload).sum() } pub fn run<R: Rng + ?Sized>(&mut self, n_ticks: u64, rng: &mut R) -> Vec<InteractionTickStats> { let mut history = Vec::with_capacity(n_ticks as usize + 1); history.push(InteractionTickStats { tick: 0, total_payload: self.total_payload() }); for tick in 1..=n_ticks { for agent in &mut self.agents { agent.advance_position(&self.motion_params, &self.bounds, self.dt, rng); } self.index.rebuild(&self.agents); step_agent_interactions(&mut self.agents, &self.index, &self.interaction, self.dt); history.push(InteractionTickStats { tick, total_payload: self.total_payload() }); } history } } #[cfg(test)] mod tests { use super::*; use rand::SeedableRng; use rand::rngs::StdRng; fn zero_motion() -> AgentParams { AgentParams { drift: [0.0, 0.0, 0.0], diffusion_coeff: 0.0, k_elim: 0.0, k_release: 0.0 } } /// Two stationary agents, always within interaction radius of each /// other: the payload trajectory should match the exact closed-form /// two-compartment exchange solution, and total payload should be /// conserved to floating-point precision at every single tick (not /// just start/end) across the whole 500-step run. #[test] fn two_agent_exchange_matches_analytical_two_compartment_solution() { let bounds = Bounds::new([-100.0, -100.0, -100.0], [100.0, 100.0, 100.0]); let agents = vec![Agent::new([0.0, 0.0, 0.0], 100.0), Agent::new([0.1, 0.0, 0.0], 20.0)]; let index = SpatialIndex::new(bounds, 4, 4, 4); let interaction = InteractionParams { radius: 5.0, k_exchange: 0.3 }; let (sum0, diff0) = (p1_0 + p2_0, p1_0 - p2_0); let t_final = 500.0_f64 * 0.01; let diff_final = diff0 * (-2.0 * 0.3 * t_final).exp(); let expected_p1 = (sum0 + diff_final) * 0.5; let expected_p2 = (sum0 - diff_final) * 0.5; If the index ever found a /// different pair set than brute force -- missed a true neighbor, /// included a false one, anything -- the payload trajectories would /// diverge. They must match to near machine precision at every tick. #[test] fn spatial_index_matches_brute_force_reference_across_full_simulation() { let bounds = Bounds::new([-6.0, -6.0, -6.0], [6.0, 6.0, 6.0]);
