Performance Profiling: Measuring Instead of Guessing

Adds phase-level timing to UnifiedEngine and runs real scaling studies across agent count and field resolution — replacing every prior "we'd need to profile before parallelizing" caveat with an actual answer.

Status: compiles clean, zero warnings. 5 new tests, all passing. Full crate: 62/62 tests passing in ~2.5s. Also run for real against workloads up to 20,000 agents — not just unit-tested in isolation.

Design: batch timing, not per-agent timing

run_profiled mirrors run()'s exact logic (same operations, same order — verified identical, see below) but wraps four points per tick in Instant::now()/elapsed(): the whole agent-update loop (motion + field sensing + decay/release), spatial index rebuild, agent-agent interaction, field diffusion.

This is deliberately coarse, and it's worth explaining why rather than assuming finer is better: a single agent's own update is a handful of arithmetic operations and an RNG draw — tens of nanoseconds, the same order of magnitude as an Instant::now() call itself. Timing at per-agent granularity would mostly measure the timer's own overhead, not the work. Batching per-phase over the whole tick keeps measured intervals large relative to timer overhead, which is what makes the numbers trustworthy.

Correctness tests

run_profiled_produces_identical_results_to_run — the load-bearing test. Two identically-built engines, identically-seeded RNGs, one run via run(), one via run_profiled(). Every tick's stats and every agent's final position/payload must match exactly. This guards against the real risk of duplicating a tick loop to add instrumentation: accidentally reordering, skipping, or duplicating an operation while doing it.

measured_phases_account_for_most_of_total_wall_time — the four measured phases plus unattributed overhead must sum to ≤100% of total wall time, and unattributed overhead must stay under 30% — a sanity check that the profiler isn't silently missing a large chunk of the real cost.

larger_agent_count_takes_longer_for_agent_update_phase / larger_grid_takes_longer_for_diffusion_phase — coarse monotonicity checks with large size gaps (50 vs 5,000 agents; 6³ vs 40³ grid) and a 5× margin, specifically to avoid flaking on ordinary timing noise.

profile_reports_csv_round_trips_row_count_and_values — same "reconstruct from the file" pattern as the output-tooling module.

Agent-agent interaction dominates overwhelmingly at scale — 96% of total time at 20,000 agents. Checked the scaling rate directly: going from 50→200 agents (4×) grows interaction cost 7.4×; 1,000→5,000 (5×) grows it 9.8×; 5,000→20,000 (4×) grows it 12.2×. This is clearly super-linear, while agent update scales cleanly linearly (3.5×–5.1× for the same 4×–5× population growth) and diffusion stays essentially flat (barely changes from 8.4 to 13.4 µs across a 400× increase in agent count, exactly as expected — diffusion cost depends only on grid cell count, not agent count).

This is not an indexing bug. The agent-count study holds the domain volume fixed while agent count grows, so agent density increases proportionally. SpatialIndex correctly reduces each individual query's cost from O(n) to roughly O(local neighbor count) — but the total interaction work summed across all agents is unavoidably driven by how many actual neighbor pairs exist within the interaction radius, and packing more agents into the same fixed volume mechanically increases that pair count faster than linearly. The index is doing its job; the workload itself gets denser.

Diffusion cost tracks grid cell count almost exactly linearly (as expected — the FTCS stencil visits every cell once per step), growing from 1.3 to 534 µs across a 512× increase in cell count — and, notably, agent update and interaction costs stay essentially flat as grid resolution changes, since neither depends on it. The two studies isolate cleanly, which is itself a useful confirmation that the profiler is measuring the right things.

What this actually tells you about parallelism

Not "parallelize the field diffusion" (the naive guess, since it's the most GPU-shaped piece of this whole crate) — at realistic agent densities, agent-agent interaction is the dominant cost by a wide margin, and it's driven by pairwise work scaling faster than linearly with density in a fixed volume. The actionable next questions this raises: does a real biological deployment actually pack agents this densely in a fixed volume, or does domain size scale with population (keeping density roughly constant, which would flatten this curve back toward linear)? That's a modeling question, not a code question — exactly the kind of finding profiling is supposed to surface before anyone writes a parallel interaction kernel for a workload that might not need one.

//! The unification: one engine running everything built so far in the //! same tick loop -- //! - motion (SDE, `sde.rs`/`agent.rs`) //! - field release + uptake (`concentration.rs`/`coupling.rs`) //! - agent-agent local exchange via the spatial index (`orchestration.rs`/`interaction.rs`) //! - field diffusion (`concentration.rs`) //! //! With /// k_elim>0 it should equal `initial_mass - cumulative_eliminated` /// exactly, since elimination is the only pathway that actually /// removes mass from the system -- every other transfer in this /// engine is a self-canceling pairwise exchange. pub total_combined_mass: f64, pub cumulative_eliminated: f64, } pub struct UnifiedEngine { pub agents: Vec<Agent>, pub params: AgentParams, pub bounds: Bounds, pub field: ConcentrationField, pub field_diffusion_coeff: f64, pub index: SpatialIndex, pub interaction: InteractionParams, pub dt: f64, } impl UnifiedEngine { pub fn new( agents: Vec<Agent>, params: AgentParams, bounds: Bounds, field: ConcentrationField, field_diffusion_coeff: f64, index: SpatialIndex, interaction: InteractionParams, dt: f64, ) -> Self { Self { agents, params, bounds, field, field_diffusion_coeff, index, interaction, dt } } fn stats(&self, tick: u64, cumulative_eliminated: f64) -> UnifiedTickStats { let total_agent_payload: f64 = self.agents.iter().map(|a| a.payload).sum(); let total_field_mass = self.field.total_mass(); UnifiedTickStats { tick, total_agent_payload, total_field_mass, total_combined_mass: total_agent_payload + total_field_mass, cumulative_eliminated, } } /// Runs the full unified tick loop.
